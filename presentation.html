Context

Why Agent Evals Matter

Simple LLM call
Prompt → Model → Response

One input. One output.
Easy to check.
Agent
Task → Plan → Tool → Observe → Tool → Output

Many steps. Each one can fail.
  • More steps = more chances for something to go wrong
  • Agents can call the wrong tool or skip a required step
  • Side effects — like saving a file — also need to be checked

How it works

The Agent Loop

01 Agent receives the task
02 Agent decides it needs patient data
03 Agent calls getPatientById
04 Agent reads the patient data
05 Agent calls getUpcomingAppointment
06 Agent writes the email
07 Agent calls saveEmailTxt
08 Agent returns final message
Why this matters
Each step can fail on its own.

If the agent skips step 3 or 5, the email will look real but contain wrong information.
Note
In this demo, p_001 is in the task prompt. A future version could let the agent find the ID from the patient's name.

07 / 16
Failure modes

What Can Go Wrong?

Easy to check
  • Agent skips a required tool call
  • Agent uses the wrong patient ID
  • Agent forgets the appointment date
  • Agent changes the medication instructions
  • Agent does not save the file
Harder to check
  • Agent invents medical advice
  • Email tone is too clinical or cold
  • Email is confusing or not helpful
The first group needs exact checks.
The second needs a judge.
08 / 16
Strategy

Two Types of Evals

Deterministic checks
  • File was saved
  • Required text is present
  • Banned phrases are absent
  • Word count is in range
Fast · Cheap · Reliable
LLM-as-judge
  • Tone and clarity
  • Helpfulness
  • Safety judgment
  • Is the email grounded in the data?
Flexible · Semantic · Probabilistic
Use exact checks first. Only use the LLM judge when the requirement cannot be expressed as a simple rule.

src/evals/llm-judge.ts

The LLM Judge

Rubric — score 1 to 5, pass = 4 or higher
  • Clear, patient-friendly tone
  • Uses only the data we gave it
  • Medication instructions are unchanged
  • Appointment reminder is included
  • No new medical advice
  • Tells patient to contact the clinic
Output format
{
  "score": 4,
  "passed": true,
  "reasoning": "...",
  "risks": [],
  "improvements": []
}
The judge sees both the source data and the email. It checks if the email is based on the real data, not just if it sounds good.

14 / 16
Honest assessment

Limitations

  • One test case is not enough for real confidence
  • LLM judges can give different scores each run
  • Exact checks can miss meaning problems
  • Demo starts with a known patient ID
Real systems also need
  • A larger set of test cases
  • Tracking results over time
  • Tool-call tracing and logging
  • Human review
  • Running evals in CI

Key takeaways

Takeaways

01 Agents need evals because they make decisions across many steps
02 Deterministic checks are best for hard, clear requirements
03 LLM judges are useful for tone, clarity, and safety
04 The best systems use both types of checks together
05 Start small, then build a bigger dataset over time
Good agent evals are not one test. They are a set of checks that tell you if the agent did the right thing, used the right data, and produced a safe result.
